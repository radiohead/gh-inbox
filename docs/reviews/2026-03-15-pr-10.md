### Medium Issues (1)

**1. JSON output omits ReviewStatus — API contract gap** `[Architecture]`
- **File:** `cmd/prs/review.go:69-81`
- **Source:** architecture reviewer
- **Issue:** The `jsonPR` struct used for `--output=json` includes `ReviewType` and `Source` (AuthorSource) but does not include `ReviewStatus`. Since the default behavior now filters to `open` only, JSON consumers cannot see why certain PRs were excluded or what status the returned PRs have. Every prior classification dimension (ReviewType, AuthorSource) is surfaced in both table and JSON output; ReviewStatus breaks this established contract.
- **Impact:** Scripts and downstream tools consuming JSON output cannot distinguish review statuses. Users who pipe `--output=json` to `jq` lose visibility into the new filtering dimension.
- **Recommendation:** Add `ReviewStatus string \`json:"reviewStatus,omitempty"\`` to the `jsonPR` struct and populate it from `cp.ReviewStatus`. This preserves the existing pattern established by ReviewType and AuthorSource.
- **Validation:** VALIDATED. `cmd/prs/review.go:69-81` exists and is confirmed in the live file. The `jsonPR` struct at lines 69-73 contains `ReviewType` and `Source` fields but no `ReviewStatus`. The file is in the diff (MODIFIED). The lines 69-81 are not directly in a changed hunk (changed hunks are 18-24, 27-33, 93-160), but this is a valid omission finding: the PR introduced `ReviewStatus` in `ClassifiedPR` (pipeline.go change) without extending the JSON output struct — a legitimate cross-cutting review point.

**1. Hardcoded `reviews(last: 20)` limit** `[Security]` `[Architecture]`
- **File:** `internal/github/queries.go:34-35`
- **Source:** security reviewer + architecture reviewer (merged duplicate)
- **Issue:** The `reviews(last: 20)` limit is a fixed constant with no runtime configurability. For PRs with extensive review cycles (common in large orgs), the last 20 reviews may not capture the full history, leading to incorrect status classification. Additionally, this data is always fetched even when `--filter-status=all` is in effect and review data is not needed, unconditionally increasing the GraphQL response payload.
- **Recommendation:** Document this as a known limitation. In a follow-up, consider making the limit configurable or skipping review fetch when the active filter would not use it.

## Strengths

- Clean orthogonal filter design: ReviewStatus added as a new independent dimension in `FilterCriteria` using the same `map[T]bool` set pattern as ReviewTypes and AuthorSources. The nil-means-match-all convention is consistent across all three dimensions.
- Correct separation of concerns: classification logic in `internal/service/review_status.go`, filter mechanics in `filter.go`, pipeline orchestration in `pipeline.go`. Domain types in `internal/github/types.go`, higher-level classification in the service layer.
- Pipeline pattern preserved: integrates cleanly into the existing fetch -> classify -> filter pipeline; `SourceClassifier.ClassifyAll` extended to call `ClassifyReviewStatus` alongside existing classifiers, keeping single-pass classification intact.
- Input validation for `--filter-status` is strict: exhaustive `switch` over the four valid values; no path where an arbitrary string flows into filter logic.
- `ReviewState` values flow exclusively from the GitHub GraphQL API through struct-based deserialization — no user-supplied strings reach review state comparisons.
- No secrets, tokens, or credentials anywhere in the diff. Authentication path unchanged.
- Classification correctly handles PENDING and DISMISSED states as non-actionable.
- Re-request heuristic is conservative and well-bounded; checked early (before team membership), preventing state confusion.
- Preset system extended consistently: focus/nearby default to `ReviewStatusOpen`; org/all leave ReviewStatuses nil.
- GraphQL extension is minimal: no new queries, no new round-trips.
- Thorough table-driven tests: 15 classification cases, 20+ `resolveFilter` cases, full preset coverage, GraphQL node conversion tests.

## Areas for Further Investigation

- Whether `reviews(last: 20)` is sufficient for repos with high review volume, and whether review fetching should be conditional on the active filter.
- Whether the `--org` query concatenation warrants a broader audit of query-builder functions.
- Whether the JSON and table output omissions (ReviewStatus not surfaced) should be treated as blocking before release given the potential for silent breakage in downstream scripts.
