# Constitution: gh-inbox

> These are invariants. Violating them requires explicit human approval.

## Project Identity

**What it is:** GitHub productivity CLI that surfaces action items across PRs, Issues, and Discussions with smart CODEOWNERS filtering

**Primary values:** correctness, minimal dependencies, composability with gh ecosystem

## Architecture Invariants

- The pipeline stages are strictly ordered: Fetch → Classify → Filter → Output.
  No stage may call a later stage directly.
- All API interaction lives in `github/`; business logic in `service/`;
  rendering in `output/`. These layers may not import each other's internals.
- `cmd/` wires the pipeline; it contains no business logic.
- Auth delegates entirely to `gh auth token` — no separate OAuth or token
  management.

**Dependency Rules**:
- Unidirectional dependency flow: cmd → (service, github, output); service → github types (interfaces only); output → service types only; github imports nothing from higher layers
- Pipeline ordering is architectural: Fetch stage may not call Classify, Filter, or Output stages; Classify may not call Filter or Output; Filter may not call Output
- ClassifiedPR and FilterCriteria flow downward through the pipeline (Classify produces them, Filter consumes them, Output renders them)
- TeamService is initialized in cmd layer and passed to SourceClassifier; no circular dependencies between components

**Structural Constraints**:
- Three-axis classification (ReviewType, AuthorSource, ReviewStatus) is immutable once computed by the service layer
- FilterCriteria uses nil-means-all semantics: omitted dimensions match all values (composable with presets and granular flags)
- The Fetcher, Classifier, and PRFilter interfaces are defined in the service package (consumer location) to decouple pipeline stages
- GitHub client wraps two narrow interfaces (graphQLDoer, restDoer) plus an optional Cacher; these are the sole testing seams

**Forbidden Patterns**:
- Service layer must not call Output functions directly (rendering is Output's responsibility only)
- GitHub client must not directly call service layer classification functions
- Output layer must not construct API requests or make GitHub calls (http or graphQL)
- Command handlers must not contain business logic—only wiring and flag parsing

## Dependency Rules

- `cmd/` may import `github/`, `service/`, `output/`, and root config.
- `service/` may import `github/` types only (interfaces, not the client).
- `output/` may import `service/` types (e.g., `ClassifiedPR`) but not `github/`.
- `github/` may not import `service/`, `output/`, or `cmd/`.
- External dependencies require a written rationale in DESIGN.md.

## Taste Rules

- Error messages are lowercase, no punctuation at end.
- No global mutable state outside of `main()`.
- Interfaces are defined where they are consumed, not where they are
  implemented.
- Fail-open for membership checks (avoids hiding PRs on transient errors);
  fail-closed for filtering (shows PR if classification is uncertain).

## Quality Standards

- All non-trivial functions have unit tests.
- Table-driven tests for all test files.
- `make build`, `make test`, `make lint` must all pass before committing.
- No linter suppressions without a comment explaining why.
